- (decision) 범위: 개인 도구 (Junghoo 혼자 사용, 팀 공유 없음) - 인증/권한 분리 불필요, 단순 CLI/스크립트 수준
- (decision) 원고 내용은 계정 불문 통일, 연락처 정보 통일 템플릿화, 홈페이지/URL 필드는 매번 수동 입력
- (decision) 작업 방식: Coaching path 선택
- (decision) PRD 진입점: Vision + Features (기능 중심, 페르소나/저니 섹션 생략 - 1인 운영 도구)
- (decision) Form-factor: 데스크탑 GUI 프로그램 (.exe 패키징, 버튼/입력창 기반) - CLI 아님, 비개발자 일상 사용 고려
- (decision) Vision 핵심: 시간절약/망각방지가 아니라 '귀찮음' 감소가 목적. 이미지 제작(품 가장 큼)은 자동화 제외, 나머지 반복작업(원고/영상/기록)의 번거로움을 줄이는 게 목표
- (decision) 이미지 제작 범위 세분화: (1)썸네일/커버 슬라이드 - 수동(일러스트레이터, 자유디자인) (2)정보카드 슬라이드(제목바+불릿박스+아이콘+HAIONNET 푸터, 레이아웃 구조는 고정이나 색상 조합은 매번 랜덤/다양하게 운용) - HTML/CSS 템플릿에 텍스트 주입 후 PNG 렌더링으로 자동화 가능, 색상은 고정값 아닌 변수 처리 필요
- (decision) 전체 작업 순서(확정): 1.썸네일 제작(수동) 2.참고이미지 보고 HTML(1080x1350, 4:5비율) 생성 3.HTML 부분수정(수동) 4.HTML to 이미지 렌더링(HTML파일/이미지파일 별도 폴더 저장) 5.HTML에 모션그래픽/애니메이션 삽입 6.5번+음악(약 20곡 플레이리스트, 이전 게시물과 안겹치게 순서대로 로테이션)으로 영상 렌더링 7.완성된 HTML 또는 이미지를 보고 SNS 원고(4사 동일) 작성 8.원고 내 홈페이지 빈칸은 수기입력 9.4사(유튜브/인스타/페북/틱톡) 수동 업로드 10.업로드 기록(엑셀)은 크롤링 제약 많아 현재는 수동 작성, 추후 별도 빌드 예정 - 이번 PRD 범위 제외
- (decision) API 사용 범위: Gemini API 키만 사용 (원고 생성용). 플랫폼별 업로드/읽기 API는 현재 범위에 불필요 (업로드 수동, 엑셀기록도 수동). API 키 등 실제 구현 단계에서 입력 예정
- (assumption) OPEN QUESTION: 원고 작성(7단계) 시 입력으로 HTML 소스 코드를 읽게 할지, 렌더링된 이미지를 읽게 할지 미결 - 사용자가 의견 요청함

- (decision) F1 세부: 정보카드 장수는 가변적(10장 이내, 보통 6장 내외, 썸네일 제외). 카드 텍스트는 별도 생성 아님 - 참고이미지 내용을 그대로 HTML로 변환(이미지to HTML 작업). 비율은 이미지/영상 모두 4:5 고정(플랫폼별 차이 없음)
- (decision) 폴더 구조: output/{키워드}/{YYMMDD}/ 안에 썸네일, html, image, video 등 해당 작업물 전부 저장
