# PRD: SNS 콘텐츠 제작 자동화 도구
*Working title — 확정 필요.*

## 0. 문서 목적
이 PRD는 1인 운영자(Junghoo)가 하이온넷(haion.net) SNS 콘텐츠 제작 과정에서 반복적이고 번거로운 작업을 줄이기 위해 만드는 개인용 데스크탑 도구의 요구사항을 정의한다. 별도 UX 문서나 선행 리서치는 없으며, 본 PRD가 유일한 기획 문서다. 기능은 카드 단위로 그룹화되고, 각 기능 아래 기능요구사항(FR)을 전역 번호로 나열한다.

## 1. Vision
SNS 콘텐츠(유튜브/인스타/페이스북/틱톡 동시 운영) 제작 과정에서 썸네일/커버 이미지(일러스트레이터 등에서 자유 디자인) 제작은 품이 가장 많이 들지만 자동화 대상이 아니다. 반면 정보카드 슬라이드 HTML 생성, 이미지/영상 렌더링, 원고 작성처럼 매번 반복되지만 창의성이 크게 필요하지 않은 작업들은 손이 많이 가고 귀찮다. [ASSUMPTION: 하이온넷은 IDC/코로케이션, 국제전용회선, 망분리, 고정IP 등 B2B 네트워크 인프라 서비스를 홍보하는 브랜드라는 전제로 본 PRD를 작성함.] 이 도구는 시간 절약이 아니라 '귀찮음 감소'를 목표로, 참고 이미지를 입력하면 HTML 카드 슬라이드 생성 → 미세 수정 → 이미지/영상 렌더링 → SNS 원고 작성까지를 하나의 데스크탑 프로그램 안에서 이어서 처리할 수 있게 한다.

## 2. Target User

### 2.1 Jobs To Be Done
- 하이온넷 SNS 콘텐츠를 혼자 기획·제작·게시하는 운영자로서, 반복 작업(카드 슬라이드 제작, 영상 렌더링, 원고 작성)에 드는 수작업 번거로움을 줄이고 싶다.
- 콘텐츠 톤/포맷의 일관성(연락처 템플릿, 카드 레이아웃 구조, 4사 동일 원고)을 매번 손으로 다시 만들지 않고 유지하고 싶다.

*단일 운영자용 내부 도구이므로 별도 페르소나 섹션과 User Journey는 생략한다(가벼운 JTBD 서술로 대체).*

## 3. Glossary
- **정보카드(슬라이드)** — 제목바, 불릿 박스, 아이콘, HAIONNET 푸터로 구성된 4:5(1080x1350) 비율의 카드 이미지. 레이아웃 구조는 고정, 색상 조합은 가변. 보통 6장 내외, 최대 10장까지 가변적으로 구성되며 썸네일은 포함하지 않는다.
- **썸네일** — 영상/게시물 맨 앞에 붙는 커버 이미지. 일러스트레이터 등에서 수동 제작하며 본 도구의 HTML화 대상이 아니다.
- **키워드** — 한 콘텐츠 묶음을 식별하는 폴더명 단위 메타데이터. 썸네일 등록 시점에 제목·홈페이지 URL과 함께 입력한다.
- **원고** — 4개 플랫폼(유튜브/인스타/페이스북/틱톡)에 공통으로 쓰이는 게시물 텍스트. 페인포인트 제시 → 핵심 포인트 이모지 불릿 → 연락처 템플릿 → 해시태그 순서를 따른다.
- **연락처 템플릿** — 모든 원고에 고정 삽입되는 문구: 대표전화 1588-1456 / 담당자 직통 010-9945-7510(09:00~18:00) / 24시 채팅상담 카카오톡 @하이온넷 / 홈페이지(URL은 썸네일 등록 시점에 입력한 폴더 메타데이터에서 자동 삽입).
- **출력 폴더** — `output/{키워드}/{YYMMDD}/` 구조로 썸네일, HTML, 이미지, 영상, 원고 텍스트 파일을 모두 보관하는 작업 단위 폴더.

## 4. Features

### 4.1 정보카드 HTML 생성
**Description:** 사용자가 참고 이미지(레이아웃/문구가 담긴 디자인 시안)를 입력하면, Claude API가 이미지 내용을 그대로 HTML(1080x1350, 4:5 비율)로 변환한다. 카드 텍스트는 별도로 새로 생성하지 않고 참고 이미지의 문구를 그대로 옮긴다. 색상 조합은 카드마다 변수로 처리되어 고정값이 아니다.

**Functional Requirements:**

#### FR-1: 참고 이미지 기반 HTML 카드 생성
사용자는 참고 이미지를 업로드하면 Claude API를 통해 동일 레이아웃 구조의 HTML 카드를 생성받을 수 있다.

**Consequences (testable):**
- 생성된 HTML은 1080x1350 해상도, 4:5 비율로 렌더링된다.
- 참고 이미지 속 텍스트의 90% 이상이 생성된 HTML에 정확히 반영된다.
- 카드 수는 1장~10장 사이에서 가변적으로 처리된다(썸네일 제외).
- 색상값은 카드마다 하드코딩이 아닌 변경 가능한 변수로 생성된다.

**Out of Scope:**
- 썸네일/커버 이미지의 HTML 변환(수동 제작 영역).

**Notes:** 실패(텍스트 반영률 미흡 등) 판단은 자동 로직 없이 사용자가 미리보기로 직접 확인 후 재생성 버튼으로 다시 요청한다.

### 4.2 HTML 편집기
**Description:** 생성된 HTML 카드를 웹뷰 기반으로 인라인 수정(contenteditable)할 수 있고, 토글로 코드창을 열어 직접 코드를 수정할 수도 있다. 미리보기에서 요소를 클릭하면 코드창의 해당 코드 위치가 하이라이트되어 두 뷰가 동기화된다. 직접 고치기 어려운 변경은 자연어로 지시해 Claude에게 맡길 수도 있다.

**Functional Requirements:**

#### FR-2: 인라인 편집
사용자는 카드 미리보기 화면에서 텍스트/요소를 직접 클릭해 내용을 수정할 수 있다.

**Consequences (testable):**
- 미리보기에서 수정한 내용은 별도 저장/반영 버튼 없이 동일 화면 갱신만으로 HTML 소스에 반영된다(키 입력 후 화면 전환 없이 확인 가능).

#### FR-3: 코드창 토글 및 동기화
사용자는 코드창을 켜고 끌 수 있으며, 미리보기에서 요소를 클릭하면 코드창의 대응 코드 라인이 하이라이트된다.

**Consequences (testable):**
- 코드창에서 직접 코드를 수정하면 별도 적용 동작 없이 미리보기 화면이 갱신된다(저장 버튼 불필요).
- 미리보기 요소 클릭 시 코드창 스크롤이 해당 코드 위치로 이동하고 하이라이트된다.

#### FR-9: 자연어 지시 기반 AI 편집
사용자는 자연어로 수정 지시(예: "이 카드 색상을 파란색으로 바꿔줘")를 입력하면 Claude API가 현재 카드 HTML을 지시대로 수정해 돌려준다.

**Consequences (testable):**
- 지시문 입력 후 결과가 적용되면 미리보기와 코드창이 함께 갱신된다.
- 수정 결과는 FR-1과 동일한 카드 골격 계약(영역 표식, 색상 CSS 변수)을 그대로 유지한다 — 구조를 벗어난 수정은 거부되고 사용자에게 알린다.
- 별도 인증/계정 연동 없이 FR-1과 동일한 Claude API 키를 사용한다.

### 4.3 이미지/영상 렌더링
**Description:** 완성된 HTML 카드를 이미지(PNG)로 렌더링하고, 모션그래픽(기본 CSS 애니메이션 수준: 페이드인/슬라이드인 등)을 삽입한 뒤 `/music/` 폴더의 음악을 순차 로테이션으로 입혀 영상으로 렌더링한다.

**Functional Requirements:**

#### FR-4: HTML → 이미지 렌더링
사용자는 HTML 카드를 1080x1350 PNG로 일괄 렌더링할 수 있다.

**Consequences (testable):**
- 렌더링된 이미지 파일은 출력 폴더의 `image/` 하위에 `{YYMMDD}_{키워드}_{카드순번 2자리}.png` 형식(예: `260625_haion망분리_01.png`)으로 저장된다. 카드 순번은 슬라이드 순서대로 01부터 부여한다.
- 원본 HTML 파일은 별도로 `html/` 하위에 동일한 순번 규칙(`{YYMMDD}_{키워드}_{카드순번 2자리}.html`)으로 보존된다.

#### FR-5: 모션그래픽 삽입
사용자는 각 카드에 기본 HTML/CSS 애니메이션(페이드인, 슬라이드인 등)을 적용할 수 있다.

**Consequences (testable):**
- 반복 모션은 최초 1회 사이클 완료를 기준으로 다음 단계로 넘어간다.
- 모든 카드는 동일한 애니메이션 패턴(페이드인 후 슬라이드인, 1회)을 사용하며, 카드별 랜덤 다양화는 적용하지 않는다. [ASSUMPTION: 구체 패턴(페이드인+슬라이드인)은 예시이며 실제 구현 시 확정 필요.]

#### FR-6: 음악 로테이션 영상 렌더링
사용자는 모션그래픽이 적용된 카드들을 음악과 함께 영상으로 렌더링할 수 있다.

**Consequences (testable):**
- 타이밍: 썸네일 5초 노출 → 각 슬라이드는 애니메이션 완료 후 3초 추가 노출 → 다음 슬라이드로 전환 → 마지막 슬라이드는 애니메이션+3초 후 영상 종료.
- 음악은 `/music/` 폴더 내 곡을 순서대로 1곡씩 사용하며, 마지막 곡 사용 후에는 처음 곡으로 순환한다.
- 마지막으로 사용한 곡 정보는 다음 실행을 위해 기록·추적된다.
- 완성된 영상 파일은 출력 폴더의 `video/` 하위에 `{YYMMDD}_{키워드}.mp4` 형식(예: `260625_haion망분리.mp4`)으로 저장된다.

### 4.4 SNS 원고 생성
**Description:** 완성된 HTML 소스와 렌더링된 이미지를 함께 Claude API에 입력하여 4개 플랫폼(유튜브/인스타/페이스북/틱톡) 공통 원고를 생성한다. 텍스트 정확도는 HTML 소스에서, 시각적 맥락과 톤은 렌더링 이미지에서 보완해 반영한다. 연락처 템플릿은 고정 삽입되고, 해시태그는 Claude가 자동 생성한다.

**Functional Requirements:**

#### FR-7: HTML+이미지 기반 원고 생성
사용자는 완성된 카드의 HTML 소스와 렌더링 이미지를 함께 입력하여 원고를 생성받을 수 있다.

**Consequences (testable):**
- 생성된 원고는 페인포인트 제시 → 핵심 포인트 이모지 불릿 → 연락처 템플릿 → 해시태그 순서를 따른다.
- 연락처 템플릿(전화/직통/카카오톡)은 매번 동일하게 자동 삽입된다.
- 홈페이지 URL은 썸네일 등록 시점에 입력된 값(§4.5 FR-8)이 자동으로 채워진다. 원고 작성 단계에서 별도로 입력받지 않는다.
- 4개 플랫폼 모두 동일한 원고 내용을 사용한다(플랫폼별 분기 없음).
- 해시태그는 별도 풀 관리 없이 Claude가 콘텐츠에 맞게 매번 생성한다.

**Out of Scope:**
- 4사 플랫폼에 대한 실제 업로드(수동 처리, 본 도구 범위 아님).

### 4.5 출력 폴더 및 메타데이터 관리
**Description:** 콘텐츠 작업물을 키워드/날짜 기준 폴더 구조로 자동 정리하고, 키워드·제목·홈페이지 URL 메타데이터를 썸네일 등록 시점에 입력받아 영상 제작 및 원고 생성(FR-7) 시 활용한다.

**Functional Requirements:**

#### FR-8: 출력 폴더 자동 생성 및 메타데이터 등록
사용자가 썸네일을 등록하면 시스템은 `output/{키워드}/{YYMMDD}/` 폴더를 생성하고, 이후 모든 산출물(썸네일, HTML, 이미지, 영상, 원고 텍스트)을 해당 폴더 하위에 저장한다. 이 등록 화면에서 키워드·제목·홈페이지 URL을 함께 입력받는다.

**Consequences (testable):**
- 동일 키워드/날짜로 재실행 시 기존 폴더에 이어서 저장된다.
- 제목 필드는 입력받아 폴더 메타데이터로 저장되며, 영상/이미지 파일명에는 반영되지 않는다(현재 활용처는 메타데이터 보존뿐, 추후 활용 예정).
- 홈페이지 URL은 입력받아 폴더 메타데이터로 저장되고, FR-7 원고 생성 시 자동으로 불러와 삽입된다.

### 4.6 제약 사항 (Constraints)
- **플랫폼**: 데스크탑 GUI 프로그램(.exe 패키징), 버튼/입력창 기반. CLI 아님 — 비개발자 일상 사용을 고려.
- **창 해상도**: 프로그램 기준 해상도 1920x1080.
- **API**: Claude API 단일 키만 사용(HTML 생성, 원고 생성 모두). 플랫폼별 업로드/조회 API는 사용하지 않음.
- **비용**: 별도 상한 없음. 1회 호출당 약 $0.16~$0.27 수준으로 부담 없는 비용으로 판단.

## 5. Non-Goals (Explicit)
- 썸네일/커버 이미지 자동 생성 — 일러스트레이터 등에서 수동 제작 유지.
- 4개 플랫폼(유튜브/인스타/페이스북/틱톡)에 대한 실제 업로드 자동화.
- 업로드 기록 엑셀 자동화/크롤링 — 크롤링 제약이 많아 현재 범위에서 제외, 추후 별도 프로젝트로 분리.
- 다중 사용자 지원, 인증/권한 분리 — 1인 운영 도구이므로 불필요.
- 플랫폼별 원고 차별화 — 4사 원고는 항상 동일 내용.

## 6. MVP Scope

### 6.1 In Scope
- 참고 이미지 → HTML 카드 생성 (Claude API)
- 웹뷰 인라인 편집 + 코드창 토글/동기화 + 자연어 지시 기반 AI 편집
- HTML → 이미지(PNG) 렌더링
- 기본 CSS 모션그래픽 삽입
- 음악 로테이션 기반 영상 렌더링

## 7. Success Metrics
*1인 내부 도구 특성상 정량 지표보다 사용 지속성과 품질 기준 중심으로 정의한다.*

**Primary**
- **SM-1**: 참고 이미지 텍스트 반영률 — 생성된 HTML 카드에 참고 이미지 텍스트가 90% 이상 정확히 반영됨. FR-1 검증.
- **SM-2**: 도구를 매 콘텐츠 제작 주기마다 실제로 사용하고 한 달 내 사용을 포기하지 않음. FR-2~FR-8 전체(편집/렌더링/원고생성/폴더관리) 검증.
- **SM-3**: 콘텐츠 1건당 HTML 카드 생성→영상 렌더링→원고 작성까지 사용자가 도구를 떠나지 않고(외부 수동 작업 도구 전환 없이) 완료할 수 있음. FR-1~FR-7 검증.

**Counter-metrics (do not optimize)**
- **SM-C1**: 재생성 버튼 클릭 횟수를 줄이는 데만 치중해 텍스트 반영 정확도(SM-1)를 희생하지 않는다.
